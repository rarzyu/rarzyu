## 1. なぜ作ったか
毎朝飲む、1日2回飲む、2日に1回飲む…など、サプリメントや薬の飲み忘れを防ぐためのシンプルなチェックリストアプリが欲しかったからです。

### 1-1. なぜ作り直したか — Flutter製旧アプリの完全リプレイス

nomow は元々「**サプリメントToDo**」というFlutter製アプリとしてリリースしていたものを、React Native + Expo で **v2.0.0として完全リプレイス** しました。

#### リプレイスを決断した理由

- **技術スタックの統一**：
  - 個人開発を TypeScript / React Native 系に集約し、メンテナンスコストを下げたかった（業務でも軸足を寄せたい技術領域）
  - また、個人的にもTypeScriptが非常に相性が良く、書いていて楽しいと感じたので統一したくなったため
- **UI / UX の全面刷新**：Flutter版はリリース後の改善が滞っており、デザインからやり直したかった
- **多言語の拡張**：2言語 → **8言語**（ja, en, zh, ko, es, fr, de, pt_BR）
- **コードベースの現代化**：Flutter版で蓄積していた技術的負債をクリーンに

#### 「使われているアプリを別言語で作り直す」判断のリスク管理

リリース済みでユーザーがいるアプリを別の技術スタックでゼロから作り直すのは、リスクの大きい意思決定です。次の方針でリスクを管理しました：

1. **既存ユーザーを置き去りにしない**：Flutter版のDB構造と互換性のあるスキーマ設計（[4.1](#41-既存ユーザーを置き去りにしない--dbスキーマのflutter互換)）
2. **Flutter現行機能をまず同等まで作る**：新機能を盛り込まず、まず移植を完了させてから機能追加（7フェーズ計画）
3. **デザインを先に固定**：実装途中で迷わないよう、Pencilで8画面を確定してから着手
4. **ストアレベルでブランド刷新**：「サプリメントToDo」→「nomow」へリブランド、What's Newで明確に告知

> この経験は、業務での **レガシーシステムリプレイス** や **段階的なフレームワーク移行** の意思決定にそのまま活かせます。


## 2. 解決したかった課題と機能の対応

### 飲み忘れの防止
サプリ・薬を飲み忘れるという課題に対し、設定した時刻に **ローカル通知** でリマインド。

### カテゴリ + カラーでの分類
種類が多くて管理しきれない課題に対し、**カテゴリ + カラー** で分類し、ドラッグ&ドロップで並び替え。

### 完全オフライン
クラウド同期不要・高速・軽量化を目指し、**端末内SQLiteのみ** で動作（サーバー送信なし・登録不要）。

### 繰り返しスケジュール（対応予定）
日替わり・週替わり・特定曜日など柔軟なスケジュール需要に対し、**繰り返し（毎日 / 毎週 / N日ごと）** を計画。

テーブル・判定ヘルパーは実装済みで、UI接続を対応予定（[4.2](#42-繰り返しを将来差し込めるテーブル設計)）。

### 8言語対応
海外ユーザーにも届けたい課題に対し、**8言語対応**（Flutter版の2言語から拡張）。

### 既存ユーザーの自然な移行
v1からv2へ自然に移行できるよう、**Flutter版互換のDBスキーマ** で設計し、ストアでは同一アプリの上書きアップデートとして配信。


## 3. 技術選定の理由

### モバイル — React Native (Expo) + TypeScript
- **理由**：
  - TypeScriptで書けてiOS・Android両対応をワンソースで実現できる
  - Expoで個人開発の運用コストを抑える
- **比較検討**：Flutter継続（技術スタック統一の方針に反する）

### DB — expo-sqlite（`openDatabaseSync` 同期API）
- **理由**：
  - 完全オフライン要件
  - Flutter版がSQLite運用だったため互換移行しやすい
- **比較検討**：
  - WatermelonDB（オーバースペック）
  - AsyncStorage（リレーションが辛い）

### 状態管理 — Zustand
- **理由**：
  - 軽量・ボイラープレートが少ない
  - Provider地獄を回避
- **比較検討**：
  - Redux Toolkit：このサイズには重い
  - React Context：再レンダ問題

### i18n — i18next + react-i18next + expo-localization
- **理由**：
  - Trecordで確立した型安全な翻訳キー戦略をそのまま流用
  - `TK.home.title` の自動補完

### 通知 — expo-notifications
- **理由**：
  - ローカル通知のみ要件
  - クロスプラットフォーム対応

### 広告 / 課金 — AdMob + `react-native-iap`
- **理由**：
  - Trecordで確立した戦略をそのまま流用

### Webサイト（LP） — Astro + Tailwind 
- **理由**：
  - LP・FAQ・お問い合わせ・利用規約・プライバシーポリシー・特商法を静的サイト（SSG・8言語）で配信 
  - Cloudflare Pagesで無料ホスティング

> 全体方針：**「Flutter版で躓いたところを学習として、TypeScript側ではきちんと再現できる」** ことを実証する個人開発として位置づけています。


## 4. 設計で意識したこと

### 4.1 既存ユーザーを置き去りにしない — DBスキーマのFlutter互換

リプレイス時に最も悩むのは「**既存ユーザーのデータをどうするか**」。

nomow ではFlutter版のDB構造と互換性のあるテーブル設計にして、将来のデータ移行パスを残しています。

- **ソフトデリート（論理削除フラグ）** を採用し、旧版のデータ表現に合わせた
- 完了状態のフラグ表現も旧版と揃え、移行時の変換を最小化
- 繰り返し機能の完了履歴は **専用テーブルに分離** し、後から差し込めるようにした（[4.2](#42-繰り返しを将来差し込めるテーブル設計)）

### 4.2 「繰り返し」を将来差し込めるテーブル設計

繰り返し機能は v2 リリース後の追加機能として計画しているため、**スキーマだけ先に用意して、ロジックは後付け** できる構造にしました。

- タスク側に繰り返し設定（毎日 / 毎週 / N日ごと）を表すフィールドをあらかじめ確保
- 日付ごとの完了履歴は **専用テーブルに分離** し、繰り返し実装時に段階的に移行できるようにした
- 現行は単一の完了フラグで管理し、繰り返し実装時に履歴テーブルへ寄せる2段階戦略

> 「**将来追加する機能の置き場だけ先に空けておく**」設計は、業務でも段階リリース・フィーチャーフラグ系の判断に直結します。

### 4.3 デザイン駆動開発：Pencilで8画面を先に確定

実装に入る前に **[Pencil](https://pencil.do/)（MCP対応の.penファイルベースのデザインツール）で8画面を確定** させました。

- カラーパレット・タイポグラフィ・コンポーネントをデザイン側で先に固める
- 実装中に「ここどう作る？」で迷う時間を最小化
- デザインと実装で同じ lucide アイコン・同じフォント（Space Grotesk / DM Sans）を使用

> 個人開発でも **デザインを先に決め切る** ことで、実装フェーズの意思決定コストを大幅に下げられました。業務でデザイナーと組む際の「Figma待ち」と同じ構造を、1人開発で再現する仕組みです。


## 5. UI / UX で意識したこと

- **登録不要・完全オフライン**：起動した瞬間からすぐ使える / プライバシー懸念を排除
- **チェックリストの単純さ**：機能を増やしすぎず、「**今日のサプリにチェックする**」体験を最短化
- **カラー + カテゴリ**で視覚的に分類：種類が多くても一覧で見分けがつく
- **ハプティクスフィードバック**：完了チェック時の触覚フィードバックで継続のモチベーション
- **ライトモード専用**（明るい、清潔感のあるヘルスケア寄りのトーン）
- **空状態（Empty State）の設計**：タスクなし・カテゴリなしの状態でも、次のアクションが分かる文言
- **8言語対応**：型安全な翻訳キーで誤キーをコンパイル時検出

- **マーケティング**：ASO分析・ストア掲載文言・施策を `docs/marketing` で管理

## 9. このプロジェクトで得た経験 → 業務での再現性

- **既存プロダクトのフルリプレイス経験**：
  - Flutter → React Native への完全リプレイスを、**既存ユーザーを置き去りにしないDB互換性** と **段階的なフェーズ計画** で進めた判断は、業務でのレガシーリプレイス案件にそのまま活かせます
- **「将来機能の置き場だけ先に切る」スキーマ設計**：
  - 繰り返し機能のためのスキーマを v1 時点で用意しておくパターンは、業務での **機能リリース計画と DB マイグレーションの整合性確保** に直結します
- **デザイン駆動開発**：
  - 1人開発でも「デザインを先に確定 → 実装は迷わない」ワークフローを再現できることを実証しました
- **パターンの再利用**：Trecordで確立した戦略を別アプリへ横展開した経験は、複数プロダクトで **共通基盤を作って横展開する** 業務スタイルに繋がります
